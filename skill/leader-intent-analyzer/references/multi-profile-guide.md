# 多画像管理指南

## 一、文件结构说明

```
leader-intent-analyzer/
├── SKILL.md                           # 核心逻辑（通用，不随画像变化）
├── references/
│   ├── behavior-patterns.md           # 行为模式库（通用）
│   ├── response-playbook.md           # 应对话术库（通用）
│   └── multi-profile-guide.md         # 本文件
├── profiles/                          # 各领导画像存储目录
│   ├── 产品组组长.md                  # 独立画像文件
│   ├── 技术总监.md                    # 另一个独立画像
│   └── [新领导].md                    # 新增画像
└── scripts/
    └── example_input.md
```

**关键设计**：
- `SKILL.md` 和 `references/` 是**通用知识库**，所有领导共享
- `profiles/` 下每个 `.md` 文件是**独立画像**，完全隔离
- 分析时只加载**当前活跃画像**，绝不混用

---

## 二、画像隔离机制

### 隔离原则
1. **素材隔离**：A领导的聊天记录绝不进入B领导的分析
2. **画像隔离**：A的特征（如"直接型"）不影响B的判断
3. **更新隔离**：更新A的画像时，B的画像完全不变
4. **记忆隔离**：每次对话只加载当前领导的画像作为上下文

### 切换机制
```
用户："切换领导：技术总监"

系统操作：
1. 卸载当前画像（产品组组长）
2. 加载 profiles/技术总监.md
3. 后续所有分析基于技术总监的画像
4. 如果技术总监不存在，提示"请先建立技术总监的画像"
```

## 三、画像迭代更新机制

### 3.1 更新触发条件

| 触发方式 | 说明 | 示例 |
|---------|------|------|
| 自动追加 | 分析意图时，新素材自动保存到素材库 | 分析完一条新消息后，素材自动归档 |
| 显式更新 | 用户说"更新画像"时全量重新分析 | "更新产品组组长的画像" |
| 矛盾标记 | 分析时发现新行为与画像不符 | 标记为"待验证"，不立即覆盖 |

### 3.2 更新流程

```
用户："这是她的新发言，帮我更新画像"
      ↓
系统：读取 profiles/产品组组长.md
      ↓
      将新素材追加到"历史素材库"顶部
      ↓
      分析新素材的行为模式
      ↓
      对比现有画像
         ├─ 一致 → 提升置信度
         ├─ 不一致 → 标记"待验证"
         └─ 多次不一致 → 触发修正
      ↓
      写入"画像变更日志"
      ↓
      保存回文件
      ↓
      输出更新摘要
```

### 3.3 更新原则

**保守修正原则**：
- 新素材与旧画像矛盾时，**优先保留旧画像**
- 新特征标记为"待验证"，等出现3次以上才正式更新
- 重大变更（如从"授权型"改"集权型"）需用户确认

**版本追溯原则**：
- 所有变更记录在"画像变更日志"
- 保留完整历史版本
- 可随时查看画像演变

**稳定度评估**：
- 素材 < 10条：低（画像可能大幅变化）
- 素材 10-30条：中（主要特征稳定）
- 素材 > 30条：高（画像成熟，仅微调）

## 四、长久记忆设计

### 4.1 记忆持久化
- 所有画像以 **Markdown文件** 形式本地存储
- 不依赖对话上下文，关闭后重新打开仍可加载
- 可手动备份、迁移、版本控制（Git）

### 4.2 记忆增量更新
- 新素材**追加**而非覆盖
- 旧素材永久保留，可追溯
- 画像分析**增量更新**，不重新分析全部历史

### 4.3 记忆检索
- 按时间检索：查看某时期的素材
- 按场景检索：查看"群里"或"私聊"的素材
- 按特征检索：查看所有"直接型"相关的素材

## 五、使用示例

### 5.1 建立多个画像

```
用户：新建领导画像：技术总监
      ↓
系统：创建 profiles/技术总监.md
      引导提供素材
      执行首次分析
      保存

用户：新建领导画像：CTO
      ↓
系统：创建 profiles/CTO.md
      ...
```

### 5.2 切换分析对象

```
用户：切换领导：技术总监
      ↓
系统：加载 profiles/技术总监.md
      当前活跃画像：技术总监

用户：他刚才说："这个方案技术风险太大，再评估一下"
      ↓
系统：基于技术总监画像分析意图...

用户：切换领导：产品组组长
      ↓
系统：加载 profiles/产品组组长.md
      当前活跃画像：产品组组长

用户：她今天安排了一个新任务...
      ↓
系统：基于产品组组长画像分析意图...
```

### 5.3 更新画像

```
用户：更新产品组组长的画像
      ↓
用户：新增素材：
      [2026-06-20 群里] "@我，这个PPT再优化一下，下午给吕毅总汇报用"
      [2026-06-21 私聊] "最近表现不错，那个手册写得很好"
      ↓
系统：追加素材 → 分析 → 对比 → 更新画像
      输出更新摘要：
      - 新增素材：2条
      - 变更发现：
        [验证] "向上管理意识"再次出现（给吕毅总汇报），置信度提升
        [新增] 发现她会给予正面反馈（"表现不错"），激励方式可能包含"认可"
      - 画像稳定度：低 → 中
```

## 六、注意事项

1. **画像命名**：建议用职位或姓名，避免重复（如"产品组组长""技术总监-张工"）
2. **素材质量**：比数量更重要，关键场景（批评、表扬、紧急任务）优先保留
3. **定期备份**：profiles/ 目录可定期复制备份，防止误删
4. **隐私安全**：所有文件本地存储，如需分享可手动删除敏感内容
5. **画像寿命**：领导风格会变化，建议每季度回顾一次画像是否需要大更新
